# Спецификация единого языка программирования

## Рабочее название: U (Universal)

Черновик v0.9 — март 2026

---

## 1. Предпосылки

Современное программирование разделено на языки не по техническим, а по социально-экономическим причинам. Все необходимые концепции уже существуют. Задача — соединить их в единый инструмент.

### 1.1. Основное противоречие

Контроль над ресурсами и удобство разработки — противоположности. Решение: четыре режима в рамках одного языка — градиент от «не думаю ни о чём» до «контролирую каждый байт».

### 1.2. Ключевое преимущество

Один язык вместо четырёх. Сейчас: Python для скриптов, Go для серверов, Rust для движков, C для embedded. Четыре экосистемы, четыре команды. U — один язык, один проект, один инструментарий. Модули с разными режимами в одном проекте.

### 1.3. Принцип проектирования

Один синтаксис. Один способ написать каждую конструкцию. Градиент от скрипта к bare metal.

## 2. Базовый фундамент

Rust: ownership, система типов, LLVM, Cargo. Реализация — транспилятор: U → Rust → rustc.

## 3. Четыре режима

### 3.1. Обзор

| Режим | Память | Загрузка | Конкурентность | Для чего |
|-------|--------|----------|----------------|----------|
| Ничего не указал | RC+GC | Вся стандартная библиотека | spawn → tokio | Скрипты, прототипы |
| `memory(auto)` | RC+GC | Только `use` | spawn → tokio | Серверы, CLI, проекты |
| `memory(own)` | Ownership | Только `use` | Thread.spawn, мьютексы | Движки, real-time, аудио |
| `memory(no)` | Ручная, без кучи | Только `use`, без рантайма | async/await bare metal, прерывания | Драйверы, embedded |

### 3.2. Градиент — от простого к сложному

Код из строгого режима всегда работает в мягком. Обратно — с ограничениями, компилятор покажет что исправить.

Рабочий процесс миграции:
1. Написал скрипт — работает, просто.
2. Вырос в проект — добавил `memory(auto)` и `use`, прозрачные зависимости.
3. Профилировщик показал горячий модуль — переключил на `memory(own)`, компилятор показал что исправить. Стало на 30-50% быстрее (нет RC overhead).
4. Нужен embedded — `memory(no)`, без кучи, без рантайма.

### 3.3. Совместимость между режимами

| Вызывающий ↓ / Вызываемый → | Скрипт | auto | own | no |
|---|---|---|---|---|
| Скрипт | ✅ | ✅ | ✅ | ✅ |
| auto | ✅ | ✅ | ✅ | ✅ |
| own | ⚠️ | ⚠️ | ✅ | ✅ |
| no | ❌ | ❌ | ⚠️ | ✅ |

✅ работает. ⚠️ работает с преобразованиями на границе. ❌ невозможно (no не может использовать кучу).

Направление снизу вверх — всегда. Модуль на `own` подключается из `auto`-проекта. Модуль на `no` — из `own`-проекта. Компилятор вставляет преобразования автоматически.

### 3.4. Конкретное преимущество каждого режима

| | Скрипт | auto | own | no |
|---|---|---|---|---|
| RC/GC overhead | Есть | Есть | Нет | Нет |
| Куча (heap) | Есть | Есть | Есть | Нет |
| Рантайм tokio | Есть | Есть | Нет | Нет |
| Свободное разделение данных | Да (RC) | Да (RC) | Нет (ownership) | Нет |
| Динамические коллекции | Да | Да | Да | Нет |
| Скорость | Базовая | Базовая | +30-50% | Максимум |

### 3.5. Примеры — один проект, четыре режима

```
// main.u — точка входа, скрипт
server = start_server()
```

```
// server.u — серверная часть
memory(auto)

use std.http: Http, Router, Response
use parser: parse_request
use firmware: read_sensor

fn handle(request)
    data = parse_request(request.body)?
    sensor = read_sensor(data.sensor_id)?
    return Response.json(sensor)
end
```

```
// parser.u — горячий парсер, без RC overhead
memory(own)

use std.io: Bytes

public fn parse_request(body: Bytes): Result[Request, Error]
    // ownership: быстро, без счётчиков ссылок
    return parse_headers(body)?
end
```

```
// firmware.u — embedded-логика
memory(no)

use hal.gpio: Pin
use hal.spi: Spi

public fn read_sensor(id: Int): Result[SensorData, Error]
    pin = Pin.open(id)?
    data = Spi.read(pin, 16)?
    return SensorData.from_bytes(data)
end
```

## 4. Мутация — оператор `::`

### 4.1. Принцип

Иммутабельность по умолчанию. `.` — чтение или команда внешней системе. `::` — изменение своих байтов в RAM. Физическая разница: кто меняет данные — мы или другая система.

### 4.2. `.` — чтение и команды внешним системам

```
users.len()              // чтение
users.first()            // чтение
db.exec("insert...")     // команда БД, db не меняется в RAM
conn.write(response)     // команда ОС отправить
ch.send("task")          // команда каналу, канал thread-safe внутри
print("hello")           // команда ОС вывести
```

### 4.3. `::` — изменение своих байтов в RAM

```
list::push(4)                        // меняет list в RAM
list::sort()                         // меняет list в RAM
user::name = "Иван"                  // меняет поле в RAM
state.player::health = health - 10   // меняет вложенное поле в RAM
cache::set("key", value)             // меняет HashMap в RAM
```

### 4.4. Цепочки

```
// чтение
result = users
    .filter(fn(x) x.age > 18)
    .sorted()
    .first()

// мутация
users
    ::sort()
    ::reverse()
    ::truncate(10)
```

### 4.5. Функции

`::` перед аргументом — в определении и при вызове:

```
fn sort(::list)
    list::sort_in_place()
end

sort(::users)         // :: при вызове — данные изменятся
process(users)        // без :: — данные не изменятся
```

### 4.6. Методы

`::self` — метод мутирующий:

```
struct List
    fn len(self): Int
        return self.items.count()
    end

    fn sort(::self)
        self.items::sort_raw()
    end
end
```

### 4.7. Переприсвоение локальных переменных

Свободное. Локальная переменная никому не видна:

```
x = 5
x = 10     // ok
```

## 5. Конкурентность

### 5.1. Скрипт и memory(auto) — spawn, tokio

`spawn` запускает задачу. Под капотом — tokio (~400 байт на задачу). Программист не видит async/await.

```
spawn handle_connection(conn)
spawn fn() send_email(user.email, "Привет")    // лямбда — одно выражение
```

Каналы:

```
ch = Channel.new()
spawn worker(ch)
ch.send("задача")
result = ch.recv()
```

### 5.2. Безопасность spawn

**Компилятор запрещает `::` на внешних данных из горутины.** `spawn writer(::data)` — ошибка компиляции. Горутина не может менять чужие байты. Чтение через `.` — разрешено.

```
data = [1, 2, 3]
spawn fn() print(data.len())     // ok: чтение
spawn fn() data::push(4)         // ошибка: мутация чужих данных
```

Мутация — через канал:

```
ch.send(data)                     // отправил копию
spawn fn()
    my_data = ch.recv()           // получил своё
    my_data::push(4)              // мутирует своё — ok
end
```

**Spawn ловит ошибки автоматически.** Ошибка через `?` → лог в stderr, горутина завершается. Паника → лог + стектрейс, горутина завершается. Остальные горутины продолжают. Разделяемое состояние не повреждено (компилятор гарантировал что горутина его не мутировала).

### 5.3. memory(own) — Thread.spawn, мьютексы

Нет tokio. Полный контроль:

```
memory(own)

use std.thread: Thread
use std.sync: Mutex

fn parallel_compute(data)
    chunks = data.split(4)
    threads = chunks.map(fn(c) Thread.spawn(fn() compute(c)))
    return threads.map(fn(t) t.join()).flatten()
end
```

### 5.4. memory(no) — async/await bare metal

Без рантайма. async/await компилируется в state machine:

```
memory(no)

use hal.timer: Timer

async fn blink(pin, interval)
    loop
        pin.toggle()
        await Timer.delay(interval)
    end
end
```

### 5.5. Параллельные коллекции — в auto и own

```
results = data.parallel().map(fn(x) heavy_compute(x)).collect()
```

Фиксированный список. Расширить нельзя.

| Модификатор | Значение | Применяется к |
|-------------|----------|---------------|
| `unsafe` | Обход проверок безопасности | Только к строке |

| `test` | Тестовая функция | К функции |
| `public` | Видимость за пределами модуля | К функции |

## 7. Синтаксис

### 7.1. Блоки — `end`

### 7.2. Спецсимволы

**Допустимые** (без переключения раскладки): `?`, `!`, `:`, `=`, `>`, `<`, `.`, `,`, `()`, `[]`, `"`, `$`, `::`.

**Запрещённые**: `{}`, `<>`, `|`, `&`, `#`, `@`, `~`, `->`, `=>`.

Логика словами: `and`, `or`, `not`.

### 7.3. Операторы

- `?` — пробрасывание ошибки / nullable тип
- `!` — force-unwrap
- `::` — мутация в RAM
- `.` — чтение / команда внешней системе
- `$` — интерполяция: `"Привет, $name!"`, `"Сумма: $(a + b)"`
- `()` — вызовы (обязательны)
- `[]` — дженерики, индексы, литералы

### 7.4. Вызовы — скобки обязательны

```
handler = process       // ссылка
result = process()      // вызов
```

### 7.5. Функции и лямбды

Именованная — стандартная форма:

```
fn add(a, b)
    return a + b
end
```

Лямбда — одно выражение:

```
data.filter(fn(x) x > 0).map(fn(x) x * 2)
```

Нужно больше одного выражения — именованная функция.

### 7.6. Return — явный

`return` = значение. Нет `return` = процедура.

### 7.7. Типизация

`public` — с типами (обязательно). Без `public` — без типов (обязательно).

### 7.8. Импорты

`use path.to.module: Name1, Name2`. Нет `*`. Обязателен только с `memory(...)`. Алиас через `as`. Неиспользованный — ошибка компиляции.

### 7.9. Базовые конструкции

```
// условия
if x > 0
    do_something()
elif x == 0
    do_other()
else
    handle()
end

// циклы
for item in collection
    process(item)
end

while condition
    step()
end

loop
    if done
        return
    end
end

// алгебраические типы
type Shape
    Circle(radius: Float)
    Rectangle(width: Float, height: Float)
end

// pattern matching
match shape
    Circle(r): pi * r * r
    Rectangle(w, h): w * h
end

// структуры
struct User
    name: String
    age: Int
end

// трейты
trait Printable
    fn to_string(self): String
end

// строковая интерполяция
greeting = "Привет, $name! Тебе $(age + 1) в следующем году."
```

## 11. Чего не будет

- Наследование классов. Трейты и композиция.
- Null/nil. `Option[T]` с сахаром `T?`.
- Исключения. `Result[T, E]` с `?`.
- Множественный синтаксис. Один способ.
- Расширяемые аннотации. 4 модификатора.
- Мутабельные переменные. Мутация через `::`.
- Многострочные лямбды. Одно выражение.
- `module` в начале файла. Имя из файла.
- Корпоративная собственность. Общественное достояние.

## 15. Рекомендации компилятору

- Арена-аллокация автоматически.
- Инлайнинг — компилятор решает.
- Hot/cold разделение.
- Escape-анализ — на стек если не покидает функцию.
- Дедупликация мономорфизации.
- В `memory(auto)`: RC по умолчанию, GC для циклов без weak, арена где возможно.
- Copy-on-write: `::` даёт точную информацию.
- Spawn → tokio: RC инкремент для чтения внешних данных, move для передачи в канал.
- Оптимизация размера задачи: tokio state machine ~400 байт vs OS thread ~1МБ.

## 16. Открытые вопросы

1. Границы между режимами — автоматические преобразования.
2. Алиасы ключевых слов на национальных языках.
3. C ABI совместимость.
4. Стандартная библиотека — состав.
5. Refinement types для own/no.
6. `!` force-unwrap — предупреждение?
7. Передача методов как значений.
8. Конкурентный доступ к структуре App из горутин.
9. Isolate (BEAM-подобная изоляция). Возможно не нужен: `::` запрещает мутацию чужих данных из горутины, spawn ловит паники — вместе дают отказоустойчивость. Нужен только если unsafe в горутине может повредить память. Проверить на практике.
10. Persistent data structures / MVCC для данных в памяти — оптимизация для случаев когда RwLock на весь объект слишком грубо.
11. Автоматический RwLock vs каналы — компилятор выбирает стратегию по `::`.
