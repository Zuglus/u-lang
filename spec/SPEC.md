## 0. Главный принцип

**Код отражает реальные связи — не придуманную архитектуру.**

### Противоречие языка

**ИИ генерирует → человек не понимает.**

Это объективное противоречие. Каждое решение в языке либо работает на его разрешение — либо лишнее.

### Метод

1. Найди противоречие в реальной задаче
2. Выведи структуру из противоречия
3. Кодируй реальные связи, не придуманную архитектуру

Не "как мне удобно" — а "как реально устроено".

## 1. Что это

U — транспилятор в Rust. Простой код на U превращается в оптимизированный Rust автоматически. ИИ генерирует U-код, транспилятор генерирует Rust, компилятор Rust создаёт нативный бинарник.

Не интерпретатор. Не виртуальная машина. Компиляция в нативный код через Rust.

### 1.1. Измерено

| Факт | Числа |
|------|-------|
| Скорость vs Python | 4.5x быстрее |
| Строк кода vs Rust | 22% меньше (137 vs 176) |
| Повторный запуск | 0.01 сек (кэш) |
| Первый запуск (hello) | ~0.5 сек (без тяжёлых зависимостей) |
| Один бинарник | Да |

> Бенчмарки vs Rust и HTTP-сервер — не верифицированы, убраны до проверки.

### 1.2. Для кого

ИИ генерирует код. Человек описывает задачу. Язык проектируется для:

- Минимум токенов → меньше ошибок ИИ
- Однозначный синтаксис → предсказуемая генерация
- Строгие правила → компилятор ловит ошибки → быстрая обратная связь ИИ
- Нативная скорость → результат сразу быстрый

### 1.3. Принципы проектирования

- Сложность от задачи, не от языка
- Каждая фича убирает конкретную проблему
- Проверяется бенчмарками, не теориями

## 3. Диалектика языка — Бытие, Ничто, Становление

Код отражает реальность. Реальность существует во времени двумя способами:

- **Нечто** (`thing`) — сохраняется во времени. Имеет идентичность, держится через моменты.
- **Процесс** (`process`) — разворачивается во времени. Идёт от входа к выходу, меняет состояние.

Нечто существует через процессы. Процессы разворачиваются над нечто. Одно без другого невозможно: нечто без процессов — мертво, процесс без нечто — не на чём действовать.

Результат процесса — **становление нового нечто**. Становление либо состоялось (Being), либо не состоялось (Nothing). Это третий момент триады.

| Реальность | В языке |
|-----------|---------|
| Нечто — сохраняется во времени | `thing` с `being/processes/nothing` |
| Процесс — разворачивается во времени | `process` с входными нечто → выход |
| Становление — стало или не стало | `Becoming[T]` с `Being/Nothing` |

Три момента жизни нечто:

- **being** — возникновение, приход в бытие
- **processes** — существование через процессы
- **nothing** — исчезновение, уход в ничто

Это то же `Being` и `Nothing` что в `Becoming[T]` — один словарь для всего языка. `Becoming[Int]` — становление Int: либо стало `Being(value)`, либо не стало `Nothing`. Гегелевская триада бытие-ничто-становление буквально зашита в систему типов.

## 4.6. Становление — `Becoming[T]`

Вместо `Option[T]` и `?` — диалектика становления. Нечто либо стало, либо не стало. Это не "возможно есть значение" (идеалистическое `Maybe`), это объективное состояние: становление состоялось или нет.

```
enum Becoming[T]
    Being(value: T)
    Nothing
end
```

`Becoming[T]` читается как "становление нечто типа T". Два исхода:
- `Being(value)` — становление состоялось, нечто родилось.
- `Nothing` — становление не состоялось, нового нечто нет. Типизировано: `Nothing[Int]` значит "не стало именно Int", контекст сохранён.

```
process divide(a: Int, b: Int) -> Becoming[Int]
    if b == 0
        return Nothing
    end
    return Being(a / b)
end

match divide(10, 2)
    Being(v) => print("стало: $v")
    Nothing => print("не стало — деление на ноль")
end
```

Отличие от `Option[T]`: `None` сообщает что нет значения вообще. `Nothing[Int]` сообщает что не стало именно Int — компилятор знает чего именно не появилось. Это важно при отладке и для ИИ-генерации: всегда понятно что ожидалось.

Гегелевская триада буквально: **бытие** (`Being`), **ничто** (`Nothing`), их единство во времени — **становление** (`Becoming`). Процесс движется от входа к выходу и производит становление нового нечто.

## 4.7. Жизнь нечто — being/processes/nothing

Нечто описывает не только форму, но и своё движение во времени — как оно существует. Существование нечто — это три момента: приход, жизнь через процессы, уход.

```
thing Server
    being: port: Int
    processes
        loop
            conn = accept(port)
            spawn handle(conn)
        end
    end
    nothing: close(port)
end

Server(8080).run()
```

`being` — приход в бытие: что нужно чтобы нечто возникло. `processes` — процессы, которые происходят пока нечто существует. `nothing` — уход в ничто: что происходит при исчезновении.

Слово `processes` названо точно: нечто живёт не само по себе, а *через процессы*, которые над ним разворачиваются. Прямо в определении нечто видно: вот процессы, из которых состоит его жизнь.

`processes` не означает цикл. Цикл — явный `loop` или `for` внутри `processes`. Без цикла — `processes` разворачивается один раз.

Компилятор выводит из жизни нечто:
- `being` → struct + конструктор
- `processes` → async метод run
- `nothing` → impl Drop

Программист не пишет `struct`, `impl`, `fn new`, `fn run`, `impl Drop` — компилятор выводит всё из being/processes/nothing. `struct/fn/impl` остаются только в сгенерированном Rust.

### Три уровня

| Уровень | Синтаксис | Пример |
|---------|-----------|--------|
| Скрипт | плоский код | `data = read_file("x")` |
| Одноразовая задача | `thing` с `being/processes/nothing` без цикла | отчёт, конвертация |
| Долгоживущее нечто | `thing` с `being/processes/nothing` и явным `loop` | сервер, воркер |

Скрипт — это неявное нечто: процесс ОС с неявным `processes` без `being` и `nothing`. Компилятор генерирует `fn main()`.

### Ограничение

`nothing` срабатывает при штатном завершении. SIGTERM не вызывает Drop в Rust — это ограничение платформы, не баг U.

### Граница с внешним миром

U видит только примитивные типы (`Int`, `String`). Rust-типы (`TcpListener`, `TcpStream`) скрыты в `.rs` модулях за `i64` handles.

```
// tcp.rs — OS граница, U сюда не смотрит
pub async fn bind(port: i64) -> i64
pub async fn accept(listener_id: i64) -> i64
pub async fn tcp_write(stream_id: i64, data: String)
```

Программист на U работает только с `Int` и `String`. Что за ними — ОС и Rust.

**Статус: реализовано, проверено на реальном HTTP сервере. `curl http://localhost:8080` возвращает ответ. 8 строк U-кода → нативный TCP сервер.**

## 4.8. Процесс — вход, разворачивание, выход

Процесс разворачивается во времени. У него нет идентичности и жизни между вызовами — есть только **определение** (рецепт, возможность) и **вызовы** (актуализация, происшествия). Между вызовами процесса в памяти не существует.

Три момента процесса во времени:
- **входные нечто** — что процесс берёт (аргументы)
- **тело** — как разворачивается (вызовы других процессов)
- **выходное нечто** — что производит (результат становления)

```
process divide(a: Int, b: Int) -> Becoming[Int]
    if b == 0
        return Nothing
    end
    return Being(a / b)
end
```

Читается: "процесс `divide` берёт два Int, разворачивается, производит становление Int — либо стало, либо не стало".

### Формула выхода

**Процесс всегда оставляет след.** Без следа процесс невозможен — его как будто и не было. След — это изменение состояния какого-то нечто. Куда направлено изменение, определяет форму выхода:

| Куда направлено | Форма в сигнатуре | Смысл |
|---|---|---|
| В новое нечто программы | `-> Becoming[T]` | Нечто типа T либо стало, либо нет |
| Во входное нечто (по ссылке) | `mut` в параметре, `-> Nothing[T]` | Новое не родилось, вход изменён |
| Во внешнее нечто (за границу) | `-> Nothing[T]` без `mut` | Новое не родилось, ушло в stdout/fs/net |

Единый словарь `Being`/`Nothing` покрывает все случаи:

```
process divide(a, b) -> Becoming[Int]       // Int может стать
process sort(mut list) -> Nothing[List]     // новый List не становится, вход меняется
process print(s: String) -> Nothing[String] // новая String не становится, ушла в stdout
process read_file(path) -> Becoming[String] // String может стать из внешнего состояния
```

`Nothing[T]` в сигнатуре выхода всегда означает одно: **нового нечто типа T не возникает**. T — тип того что было бы результатом, если бы результат рождался. Для `sort` это List (естественный результат — отсортированный список, но он тот же самый). Для `print` это String (то что мы отдали в мир). Для `divide` при делении на ноль — Int (тот что не родился).

### Определение процесса всегда в `.u`

Процесс определяется в файле `.u` — всегда, в том числе ядерные процессы. Компилятор не знает специальных имён, никакого хардкода. `print`, `read_file`, `sleep` — это обычные `process`-определения в `core/*.u`, тело которых зовёт Rust-обёртки из соседних `core/*.rs` файлов:

```
// core/io.u
use rust_io: println_raw

process print(s: String) -> Nothing[String]
    println_raw(s)
end
```

```rust
// core/io.rs
pub fn println_raw(s: String) {
    println!("{}", s);
}
```

Программист открывает `core/io.u`, видит определение `print` — никакой магии. Всё открыто. Граница между U и Rust видна: тут `.u` с `process`, там `.rs` с syscall.

Правило: **процесс определяется либо в `.u`, либо в `.rs` (через `use`). Третьего пути нет.** Компилятор генерирует вызов одинаково независимо от уровня доступности.

### Строковая интерполяция — свойство литерала, не процесса

`"hello, $name"` на уровне парсера всегда превращается в `format!("hello, {}", name)` — независимо от того, куда передаётся результат. Интерполяция принадлежит строковому литералу, а не вызову `print`. Это обеспечивает "один способ": любая U-строка с `$name` ведёт себя одинаково всюду.

## 5. Мутация — `mut`

### 5.1. Правило

`mut` в определении процесса — единственный маркер мутации. При вызове — всегда `.`. Компилятор генерирует `&` или `&mut` автоматически.

### 5.2. Процессы

```
// чтение — без mut
process total(list) -> Int
    return list.sum()
end

// мутация — mut в параметре
process sort(mut list)
    list.sort_in_place()
end

// вызов — всегда .
print(total(data))
sort(data)
```

### 5.3. Процессы с `self`

```
impl Counter
    process get(self) -> Int
        return self.value
    end

    process increment(mut self)
        self.value = self.value + 1
    end
end

c.get()
c.increment()
```

### 5.4. Генерируемый Rust

| U-код | Rust |
|-------|------|
| `process show(user)` | `fn show(user: &User)` |
| `process sort(mut list)` | `fn sort(list: &mut Vec<i64>)` |
| `show(user)` | `show(&user)` |
| `sort(data)` | `sort(&mut data)` |
| `process get(self)` | `fn get(&self)` |
| `process increment(mut self)` | `fn increment(&mut self)` |

### 5.5. Субъект действия

Все процессы — одна сущность. Различие в субъекте:

- Процесс с `self`, вызывается через `.` — нечто действует на свои данные: `text.trim()`, `counter.increment()`.
- Процесс без `self` — внешнее действие над входным нечто: `print(text)`, `sleep(100)`.

Оба — `process`, один механизм определения, один механизм вызова. Точка перед именем значит "субъект — то что слева".

## 6. Конкурентность

### 6.1. Spawn

```
spawn handle(conn)

process worker(ch)
    loop
        task = ch.recv()
        handle(task)
    end
end
spawn worker(ch)
```

### 6.2. Каналы

```
use std.channel: Channel

ch = Channel.new()
ch.send("задача")
result = ch.recv()
```

### 6.3. Правила spawn

- Внешние переменные видны на чтение
- Мутация внешних данных в spawn запрещена (компилятор проверяет по `mut`)
- Spawn ловит паники автоматически

```
data = [1, 2, 3]
spawn
    print(data.len())     // ok — чтение
end

spawn
    sort(data)            // ошибка: sort(mut list) — мутация внешнего
end
```

### 6.4. Автопараллельность for (исследуется)

Компилятор анализирует тело `for`: нет `mut` на внешних данных → итерации независимы → может параллелить автоматически. Решение по практике.

### 6.5. Реализация

`spawn` → `tokio::spawn` (~400 байт на задачу). Каналы → `tokio::sync::mpsc`.

## 7. Синтаксис

### 7.1. Как в Rust

```
trait, impl, enum, pub, match + =>,
for, while, loop, if, else, elif, //, ->
```

U заменяет `fn` на `process` (действие во времени) и `struct` на `thing` (нечто с жизнью).

### 7.2. Отличия от Rust

| U | Rust | Причина |
|---|------|---------|
| `end` | `}` | Надёжнее для ИИ-генерации |
| нет `;` | `;` | Меньше токенов |
| `x => expr` | `\|x\| expr` | Анонимные процессы — без ключевого слова, читаются слева направо |
| `(x, y) => expr` | `\|x, y\| expr` | Несколько параметров |
| `$name` | `format!("{name}")` | Меньше токенов |
| `mut` в сигнатуре | `&mut` при вызове | Проще — маркер только в определении |
| нет `let` | `let`/`let mut` | Меньше токенов |
| `and`, `or`, `not` | `&&`, `\|\|`, `!` | Читаемость |
| `[T]` дженерики | `<T>` | Клавиатура |
| `spawn f()` | `tokio::spawn(...)` | Одно слово |
| `return` явный | последнее выражение | Один способ |
| скрипт без main | `fn main()` | Меньше boilerplate |

```
map["key"]  map.len()  map.keys()

// Int
n.abs()  n.to_string()
```

### 7.9. Циклы

| Конструкция | Для чего |
|-------------|---------|
| `for item in list` | Итерация по коллекции |
| `while condition` | Цикл с условием |
| `loop` | Бесконечный цикл (серверы, воркеры) |
| `break` | Выход из цикла |
| `continue` | Пропуск итерации |

## 8. Интеграция с Rust

`.rs` файлы — полноценные модули в проекте:

```rust
// fast_math.rs
pub fn fibonacci(n: u64) -> u64 {
    if n <= 1 { return n; }
    let (mut a, mut b) = (0u64, 1u64);
    for _ in 2..=n { let tmp = a + b; a = b; b = tmp; }
    b
}
```

```
// main.u
use fast_math: fibonacci
print("fib(50) = $(fibonacci(50))")
```

U для 90% кода. Rust для 10% — парсеры, драйверы, SIMD, криптография.

## 9. Стратегия памяти

### 9.1. Принцип

Программист не думает о памяти. Компилятор выбирает оптимальный механизм. `mut` даёт информацию.

### 9.2. Что работает сейчас

- Stack-first — структуры на стеке
- Borrowing — нет `mut` → `&`, есть `mut` → `&mut`
- Move — последнее использование → move

### 9.3. Исследовано, не реализовано

| Механизм | Источник |
|----------|---------|
| Escape-анализ | Go, Java JIT |
| Reuse / FBIP | Perceus (Koka) |
| Compile-time RC elision | Lobster |
| Arena | Verona |
| COW | Swift |
| Auto-weak | Новое |

Реализация — когда появится конкретная задача.

## 10. Решаемые проблемы

### 10.1. Проблемы Rust

| Проблема | Решение U |
|----------|-----------|
| Сложный синтаксис (lifetimes, `&mut`) | `mut` в сигнатуре, компилятор генерирует |
| Медленная компиляция | Кэш, умные зависимости |
| Крутая кривая обучения | Скрипт без main/use/типов |
| Boilerplate | Нет `let`, `;`, `fn main()` |

### 10.2. Проблемы Python

| Проблема | Решение U |
|----------|-----------|
| Медленный (4.5x) | Нативная компиляция через Rust |
| GIL — нет параллельности | spawn, реальные потоки |
| None → крэш в рантайме | Becoming[T] — Being/Nothing при компиляции |
| pip hell, деплой | Один бинарник |

### 10.3. Проблемы Go

| Проблема | Решение U |
|----------|-----------|
| GC паузы | Нет глобального GC |
| if err != nil | Result[T, E] с явной обработкой |
| Нет enum/ADT | enum + match => |
| Нет доступа к Rust-экосистеме | .rs файлы, crates.io |

## 11. Чего не будет

- Интерпретатор. Только компиляция.
- Наследование. Трейты и композиция.
- Null/nil. `Becoming[T]` с `Being/Nothing`.
- Исключения. `Result[T, E]`.
- Множественный синтаксис. Один способ.
- Глобальный GC.
- Виртуальная машина / байткод.
- Extension functions. Методы через `impl`.
- Перегрузка процессов. Уникальные имена.
- Implicit conversions. Явные преобразования.
- Макросы. Явный код.
- Глобальное mutable состояние.

## 14. Открытые вопросы

1. Автопараллельность `for` — проверить практикой.
2. Генератор — Nothing, .first(), .filter(), JSON field access.
3. Стандартная библиотека — HTTP-клиент, расширение.

8. **Главное противоречие (не решено):** миссия говорит "программист не думает о памяти и потоках" — но разделы 5 (mut), 6 (spawn), 9 (стратегия памяти) описывают именно это. Как скрыть управление памятью и конкурентностью полностью, сохранив гарантии Rust — открытый вопрос. Исследуется.

Миссия: программист не думает о памяти, потоках, синхронизации.

Реальность: `mut`, `spawn`, стратегия памяти — всё это сейчас явно в языке и требует понимания от программиста.

Вопрос: как скрыть управление памятью и конкурентностью не потеряв гарантии безопасности?
